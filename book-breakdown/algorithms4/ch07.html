两摞已排序的扑克牌
左手一摞、右手一摞，都已从小到大排好 每次比较两摞最上面的牌，取较小的放到桌上 任何一摞用完就把另一摞全部放下 结果：桌上就是完整的有序牌堆

💻
merge(a, lo, mid, hi)
a[lo..mid] 和 a[mid+1..hi] 各自有序 用双指针 i、j 分别扫描两半，取较小者放入结果 某一半用完就把另一半剩余元素依次放入 结果：a[lo..hi] 整体有序
归并排序的代码演进

从最基本的抽象归并到自顶向下、自底向上，再到三种实用优化——每一步都在同一个 merge() 基础上构建。

自顶向下 vs 自底向上
自顶向下 (Top-down)
sort(a, 0, 15) sort(a, 0, 7) sort(a, 0, 3) merge(a, 0, 0, 1) // 先处理最小块 merge(a, 2, 2, 3) merge(a, 0, 1, 3) // 再合并 ... merge(a, 0, 3, 7) // 最后大合并 ... merge(a, 0, 7, 15) // 最终合并
VS
自底向上 (Bottom-up)
// sz=1: 两两合并 merge(a, 0,0,1) merge(a, 2,2,3) ... // sz=2: 四个一组合并 merge(a, 0,1,3) merge(a, 4,5,7) ... // sz=4: 八个一组合并 merge(a, 0,3,7) merge(a, 8,11,15) // sz=8: 最终合并 merge(a, 0,7,15)
当 N 是 2 的幂时，两者执行完全相同的比较和数组访问，只是顺序不同。 自顶向下用递归（更自然），自底向上用迭代（更适合链表）。 核心洞察：分治可以「自上而下拆」也可以「自下而上建」。
🔒
任何基于比较的排序算法，最坏情况下至少需要 lg(N!) ~ N lg N 次比较。 证明思路（决策树）：N 个元素有 N! 种排列，每次比较最多排除一半可能性，因此至少需要 lg(N!) 次比较才能确定是哪种排列。 这意味着归并排序是渐近最优的——你不可能发明一个比较排序算法，在最坏情况下比 N lg N 快一个常数倍以上。这就像光速是信息传递的极限一样，N lg N 是比较排序的极限。
