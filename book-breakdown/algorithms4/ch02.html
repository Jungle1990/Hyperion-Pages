从使用到创建：Counter ADT 的完整旅程

通过 Counter 这个最简单的 ADT 示例，展示从 API 定义到客户端使用到内部实现的全过程。

别名陷阱：引用 vs 值的致命区别
基本类型（复制值）
int x = 5; int y = x; // y 复制了值 5 y = 10; // 只改了 y // x 仍然是 5 ✓ 符合直觉
VS
引用类型（复制引用）
Counter c1 = new Counter("ones"); c1.increment(); Counter c2 = c1; // c2 复制了引用! c2.increment(); // c1 和 c2 指向同一对象 // c1 的值也变成了 2! ⚠ 别名陷阱
这是对象编程中最常见的 bug 来源。c2 = c1 不是创建了一个新的 Counter，而是让 c2 和 c1 指向「同一个」Counter 对象。修改一个，两个都变。这种现象叫「别名」(aliasing)。 对比：Date newDate = new Date(1, 1, 2024) 才是创建新对象。
同一个 API，两种内部实现
直觉版：三个字段
public class Date { private final int month; private final int day; private final int year; public Date(int m, int d, int y) { month = m; day = d; year = y; } public int month() { return month; } public int day() { return day; } // 空间: 3 个 int = 12 字节 // 时间: 直接返回, O(1) }
VS
紧凑版：一个字段
public class Date { private final int value; public Date(int m, int d, int y) { value = y*512 + m*32 + d; } public int month() { return (value / 32) % 16; } public int day() { return value % 32; } // 空间: 1 个 int = 4 字节 // 时间: 需要算术运算 }
这就是封装的威力：客户端代码完全不变！无论内部用 3 个 int 还是 1 个 int，StdOut.println(date) 的输出都是 12/31/1999。 我们可以随时把一种实现换成另一种，不需要修改任何客户端代码。这正是「数据抽象」的核心价值。
🛡️
封装不只是「把变量设为 private」，它带来四重保护： 1. 独立开发：客户端和实现可以由不同的人分别开发，只要遵守 API 合同 2. 无痛升级：可以替换更好的实现而不影响任何客户端代码（如 Date 的两种实现） 3. 前向兼容：API 就是未来客户端的指南，支持尚未编写的程序 4. 错误隔离：bug 的影响范围被限制在单个模块内，不会扩散 正如书中所说：「你不需要知道数据类型如何实现就能使用它，你也不需要知道客户端如何使用就能实现一个数据类型。」
