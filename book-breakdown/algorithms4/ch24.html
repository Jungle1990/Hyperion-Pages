核心类比：数据压缩如同「拧毛巾」
🧺
拧干毛巾
毛巾吸满水 = 原始数据充满冗余 拧的力度 = 压缩算法的选择 不同材质需要不同拧法 = 不同数据结构需要不同压缩策略 拧干后重新泡水能恢复 = 无损压缩可以完美还原
⟺
💾
压缩比特流
原始比特流 = 充满冗余的数据 游程编码 = 合并连续重复的比特 Huffman = 给高频字符分配短编码 LZW = 发现重复出现的子串并建立词典 解压后完全还原原始数据 = 无损压缩
🚫
没有任何算法能压缩每一个比特流！ 证明一（反证法）：如果能压缩每个比特流，就能反复压缩直到 0 位——荒谬。 证明二（计数论证）：2^1000 个 1000 位串要映射到不足 2^1000 个更短的串，根据鸽巢原理不可能一一映射。 实际影响：随机数据无法被压缩。但我们处理的数据几乎都有结构，这正是压缩算法大展身手的舞台。
三大压缩算法的演进

从最简单的游程编码到自适应的 LZW，每种算法利用数据中不同类型的冗余。

Huffman vs LZW：两种互补的策略
Huffman（变长编码 + 定长模式）
输入: 定长字符（8位） 输出: 变长编码 核心: 高频字符 → 短编码 需要: 两遍扫描（先统计频率） 必须: 将 Trie 写入压缩流 代表: zip/gzip 的基础组件
VS
LZW（定长编码 + 变长模式）
输入: 变长子串（最长前缀匹配） 输出: 定长编码（如12位） 核心: 长子串 → 一个编码 需要: 一遍扫描（自适应） 无需: 传输词典（解压端可重建） 代表: gif/compress 的核心
Huffman 在《双城记》上压缩率 52%，LZW 则达到 46%。LZW 更擅长利用重复出现的长子串（如自然语言文本中的常用词组），而 Huffman 在字符频率差异大时最有效。两者常被组合使用——先 LZW 再 Huffman。

LZW 解压有一个微妙的边界情况：当压缩端刚添加的编码恰好是下一个要使用的编码时，解压端还没来得及建立这个条目！ 例如输入 ABABABA，压缩输出 41 42 81 83 80。解压到编码 83 时，解压端需要查表得到前瞻字符来完成条目 83，但条目 83 正是当前要建立的那个！ 解决方案：检测到这种情况时（codeword == i），用当前字符串的第一个字符作为前瞻字符。这种情况恰好发生在模式 cScSc 中（c 是字符，S 是字符串），此时 cS 已在词典中但 cSc 不在。
