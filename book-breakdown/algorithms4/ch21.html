在 Trie 中搜索
从根节点出发，跟随第一个字符的链接 逐字符向下移动 到达终点节点就返回关联的值 遇到 null 链接就返回未找到 搜索时间正比于键的长度

Trie 的搜索与插入
搜索命中 (Search Hit)
get("shells") root -> s -> h -> e -> l -> l -> s // 到达最后一个字符对应的节点 // 节点的 val != null // 返回该值 (val = 3) get("she") root -> s -> h -> e // 搜索可能终止在内部节点 // val != null, 返回 val = 0
VS
搜索未命中 (Search Miss)
get("shell") root -> s -> h -> e -> l -> l // 到达最后一个字符对应的节点 // 但 val == null // 返回 null（键不在表中） get("shore") root -> s -> h -> o -> null link! // 遇到 null 链接 // 直接返回 null
搜索命中的时间正比于键的长度（命题 G）。搜索未命中平均只需检查约 logR(N) 个节点（命题 H），对于 R=256 和 N=100万，这意味着只需检查约 3-4 个字符！

R-way Trie vs TST：时空权衡
R-way Trie
// 每个节点 R 个链接 Node[] next = new Node[R]; // R=256 // 优点：搜索极快，logR(N) 次字符比较 // 缺点：空间 RN ~ RNw 个链接 // 搜索未命中：~logR(N) 次字符检查 // 搜索命中：最多 键长+1 次数组访问 // 适合：短键 + 小字母表
VS
三向单词查找树 (TST)
// 每个节点只有 3 个链接 Node left, mid, right; char c; // 显式存字符 // 优点：空间 3N ~ 3Nw 个链接 // 缺点：搜索多一个 lnR 倍因子 // 搜索未命中：~lnN 次字符比较 // 搜索命中：每个字符约 1 次比较 // 适合：大字母表 + 长键
R-way Trie 相当于对每个节点用「键索引数组」，TST 相当于对每个节点用「二叉搜索树」。TST 对应三向字符串快速排序，正如 BST 对应快速排序。
