核心类比：归并排序 vs 快速排序
🔀
归并排序 (Mergesort)
像拆快递：先把大箱子拆成小箱子 分别整理好后再合并（merge） 切分位置固定（总是对半分） 稳定但需要额外空间
⟺
⚡
快速排序 (Quicksort)
像排队量身高：先找一个标杆就位 标杆左边都比它矮，右边都比它高 切分位置取决于数据内容 原地排序，内循环极短
Partition：快速排序的灵魂操作

partition() 是快速排序的核心——每调用一次，就把一个元素「钉死」在最终位置。理解它的演进就理解了整个算法。

快速排序性能对比（排序 100 万个元素）

🧠
归并排序的 N log N 下界是针对「所有键都不同」的最坏情况。当存在大量重复键时，数组的信息含量（Shannon 熵 H）远低于 lg N。三路快排的比较次数正比于 NH，达到信息论下界的常数倍——这就是「熵最优」的含义。 举例：100 万条记录只有 3 种取值时，H ≈ 1.58（而非 lg 1000000 ≈ 20），三路快排接近线性时间完成排序！
快速排序的五个致命陷阱
正确做法
// 1. 排序前随机打乱 StdRandom.shuffle(a); // 2. 遇到等于 pivot 的元素，停下来交换 while (less(a[++i], v)) ... while (less(v, a[--j])) ... // 3. 指针交叉时正确终止 if (i >= j) break; // 4. 最后将 pivot 与 a[j] 交换 exch(a, lo, j); // 5. 小数组切换插入排序 if (hi <= lo + M) Insertion.sort(...);
VS
常见错误
// 1. 不打乱 → 有序输入退化 O(N²) // 跳过 shuffle // 2. 跳过等于的元素 → 全等数组退化 O(N²) while (a[++i] < v) ... while (a[--j] > v) ... // 3. 指针交叉判断出错 → 无限循环 if (i > j) break; // 漏了 i==j // 4. 最终交换对象搞错 → 排序结果错误 exch(a, lo, i); // 应该是 j！ // 5. 不处理小数组 → 大量递归开销 // 一直递归到 size=1
快速排序的「脆弱性」在于：每个细节都可能导致性能从 N log N 退化到 N²。Sedgewick 在书中反复强调这些教训来自数十年的实践错误。
