散列表
散列函数 → 映射到数组下标 不同的键可能映射到同一下标（碰撞） 负载因子过高就需要 resize 扩容 查找/插入期望 O(1) 时间
散列函数的设计演进

从简单取模到 Java hashCode()，散列函数的设计目标始终是：一致性、高效、均匀分布。

两大碰撞解决策略对比
拉链法 (Separate Chaining)
public class SeparateChainingHashST { private int M; // 散列表大小 private SequentialSearchST[] st; public Value get(Key key) { return st[hash(key)].get(key); } public void put(Key key, Value val) { st[hash(key)].put(key, val); } } // 每个桶是一条链表，碰撞的键挂在同一条链上
VS
线性探测法 (Linear Probing)
public class LinearProbingHashST { private Key[] keys; private Value[] vals; private int M; public void put(Key key, Value val) { int i; for (i = hash(key); keys[i] != null; i = (i + 1) % M) if (keys[i].equals(key)) { vals[i] = val; return; } keys[i] = key; vals[i] = val; } } // 碰撞时向后探测，找到空位插入
拉链法：实现简单、删除容易、负载因子可 > 1，但需要额外链表内存。 线性探测法：缓存友好（连续内存）、空间紧凑，但负载因子必须 < 1，删除需要重新插入后续键，且会出现「聚集」现象。
⚠️
线性探测中，连续被占用的位置形成「簇」(cluster)。长簇就像高速公路上的连环追尾——新车（键）撞上车队尾部后，车队变得更长，后面的车更容易撞上来。 Knuth 在 1962 年证明：当负载因子 α = N/M 时，搜索命中平均探测次数约为 ½(1 + 1/(1-α))，搜索未命中约为 ½(1 + 1/(1-α)²)。 当 α = 1/2 时：命中 ~1.5 次，未命中 ~2.5 次 —— 很快！ 当 α = 3/4 时：命中 ~2.5 次，未命中 ~8.5 次 —— 已经明显变慢。 当 α → 1 时：探测次数趋向无穷 —— 完全不可用！ 因此必须通过数组扩容保持 α ≤ 1/2。
线性探测：不同负载因子下的平均探测次数
