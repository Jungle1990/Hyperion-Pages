数组排序
选择排序：每轮扫描未排序部分，找最小值交换到前面 插入排序：维护左侧有序区，新元素逐个比较插入 希尔排序：先以大步长 h 做插入排序，逐步缩小 h 到 1
📏
排序算法的全部操作只依赖两个原语：less()（比较）和 exch()（交换）。Java 通过 Comparable 接口将「比较」抽象化 -- v.compareTo(w) 返回负数、零或正数，分别表示 vw。 compareTo() 必须满足全序关系的三条公理：自反性（v=v）、反对称性（vv）、传递性（v<=w 且 w<=x 则 v<=x）。这三条看似显然，却是排序正确性的数学基石。
三种排序算法的代码演进

从最简单的选择排序出发，到感知输入的插入排序，再到突破二次复杂度壁垒的希尔排序。

选择排序 vs 插入排序：对输入的敏感度
选择排序
// 已排序数组 [1,2,3,4,5] // 选择排序仍需 N(N-1)/2 次比较 // 因为它必须扫描整个未排序区 // 才能「确认」最小值 // 运行时间 ≈ N^2/2 (不变!)
VS
插入排序
// 已排序数组 [1,2,3,4,5] // 插入排序只需 N-1 次比较 // 每个元素发现左邻 <= 自己 // 立即停止，不再向左探索 // 运行时间 ≈ N (线性!)
选择排序的优势在于「数据移动最少」-- 恰好 N 次交换，适合交换成本远高于比较成本的场景（比如搬运大型箱子）。插入排序的优势在于「适应性」-- 输入越有序，速度越快，适合数据「几乎有序」的场景。

希尔排序用一个看似微小的修改 -- 把步长从 1 变成递减序列 h -- 就打破了初级排序 O(N^2) 的壁垒。 为什么有效？因为 h-排序有一个关键性质：一个 h-有序的数组在 k-排序后仍然保持 h-有序。这意味着每一轮排序都在前一轮的基础上「锦上添花」，而不是推倒重来。 递增序列 1, 4, 13, 40, 121... (3x+1) 的最坏情况是 O(N^{3/2})，比 N^2 好得多。但至今没有人证明什么是最优递增序列 -- 这是一个超过 60 年的未解问题。
